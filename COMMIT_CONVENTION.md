# 提交规范

## 格式化提交信息

提交信息应遵循以下格式：

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

其中，第一行 `<type>(<scope>): <subject>` 是必需的。

## 类型 (Type)

提交的类型必须是以下之一：

- `feat`: 新功能
- `fix`: 修复错误
- `docs`: 文档更新
- `style`: 代码格式调整，不影响逻辑（空格、格式、缺少分号等）
- `refactor`: 重构代码，既不修复bug也不添加功能
- `perf`: 性能优化
- `test`: 添加缺失测试或修正现有测试
- `chore`: 构建过程或其他杂项维护任务

## 范围 (Scope)

范围是可选的，表示提交影响的功能模块。例如：

- feat(api): 添加新API
- fix(parser): 修复解析器错误
- docs(readme): 更新README文档

## 主题 (Subject)

主题应简洁明了地描述变更内容。需注意：

- 使用祈使句，例如 "change" 而不是 "changed" 或 "changes"
- 不要首字母大写
- 不要在末尾加句号

## 正文 (Body)

正文应详细说明变更的原因和内容：

- 使用祈使句
- 可以解释为什么这个变化是必要的
- 描述变更的影响

## 脚注 (Footer)

如果提交包含破坏性变更，在脚注中说明：

```
BREAKING CHANGE: 详细描述变更及其影响
```

## 示例

```
feat(api): 添加用户认证接口

添加登录、注册和登出API接口，支持JWT令牌验证。

- 登录接口验证用户凭据并返回JWT令牌
- 注册接口创建新用户并发送确认邮件
- 登出接口使当前会话失效

Closes #123
```

```
fix(database): 修复数据库连接泄漏

在数据库查询完成后确保连接正确关闭，防止长时间运行的应用出现连接池耗尽的问题。

Fixes #456
```
